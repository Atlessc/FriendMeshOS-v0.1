# FriendMesh Decision Log

Status: active Phase 0 control document  
Last reviewed: 2026-07-12  
Decision owner: FriendMeshOS product owner  
Maintainers: firmware, protocol, security, storage, and T-Deck UI owners

This log freezes the approved product and architecture choices that implementation must preserve. It is a tracker, not a substitute for the full contracts in [`FRIENDMESH_FULL_IMPLEMENTATION_GAME_PLAN.md`](../../FRIENDMESH_FULL_IMPLEMENTATION_GAME_PLAN.md) and [`MESHTASTIC_CORE_AND_FRIENDMESH_ARCHITECTURE.md`](../../MESHTASTIC_CORE_AND_FRIENDMESH_ARCHITECTURE.md).

## Authority and change control

When sources disagree, use this order:

1. Compiled source for facts about behavior that already exists.
2. The full implementation game plan for approved FriendMesh product behavior.
3. The architecture handbook for the Meshtastic compatibility boundary.
4. This log for the status, rationale, owner, and evidence trail of a choice.

A decision may change only through a new decision record. Never silently rewrite the original row. Mark the old row `SUPERSEDED`, add the replacement ID, describe migration and wire/storage compatibility effects, update the canonical contracts, and attach test or review evidence.

Status values:

- `APPROVED`: binding for implementation.
- `RESEARCH`: implementation choice remains open, but the product boundary is fixed.
- `BLOCKED`: cannot be finalized without named evidence or authority.
- `SUPERSEDED`: retained for history; a newer record controls.
- `REJECTED`: considered and deliberately not selected.

Evidence values should be links to committed design notes, tests, captures, reports, or release artifacts. `Planned` is not completion evidence.

## Decision review checklist

- [ ] Affected phases, screens, and test campaigns are named.
- [ ] Canonical game plan and architecture handbook are reconciled.
- [ ] Risk-register entries are added or updated.
- [ ] Upstream-delta ledger is updated when a Meshtastic boundary changes.
- [ ] Evidence is committed and linked; planned work is not marked verified.
